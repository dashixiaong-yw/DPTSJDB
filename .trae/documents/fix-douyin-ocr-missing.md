# 抖店OCR识别结果缺失 — 根因分析与修复计划

## 问题描述

用户反馈：抖店账单的对比结果中，所有OCR识别数据缺失/全为0。

## 根因分析

经过详细测试和日志分析，发现以下**3个独立问题**共同导致结果页数据缺失：

### Bug 1: Qwen3-VL模型对部分截图识别不准确（核心问题）

**现象**：LLM API返回了短响应（108字符），将金额全部设为0，但店铺名称和月份识别正确。

**证据**：
- 服务器日志显示行8（指尖时光）的店铺月度数据截图OCR结果：`{"成交金额":0,"退款金额":0,"投放消耗":0}`
- 单独测试image13（644KB，L8列）时，OCR也返回全0
- 单独测试image12（0.8KB，N7列）时，OCR也返回全0（这是异常图片，只有802字节）
- 但image1（梵仔，477KB）的OCR结果是正确的：`{"成交金额":115183.3,"退款金额":20584.44,"投放消耗":0}`

**根因**：`Qwen/Qwen3-VL-32B-Instruct` 模型对某些截图的识别能力不足，特别是：
1. 图片内容较复杂时（如指尖时光的截图），模型倾向于返回简短响应并将金额设为0
2. 异常图片（如image12只有802字节）无法识别
3. 提示词可能需要针对Qwen模型优化（原提示词是为Kimi模型设计的）

### Bug 2: 缓存完整性检查过于严格（次要问题）

**现象**：`isCacheComplete` 对抖音店铺月度数据截图走默认分支，要求 `shop_name + month + amounts` 全部存在。

**证据**：
- 当OCR返回 `shop_name="指尖时光好物", month="5月", amounts={"成交金额":0}` 时，缓存被标记为完整
- 但当OCR返回 `shop_name="指尖时光好物", month="", amounts={"成交金额":115183.3}` 时，缓存被标记为不完整
- 这导致缓存反复重新识别，但每次结果可能不同

**根因**：`isCacheComplete` 没有为抖音店铺月度数据截图设置专门的完整性检查规则（只设置了支出总额截图的规则）。

### Bug 3: image12（N7列）是异常图片（数据问题）

**现象**：image12只有802字节，OCR识别全为0。

**证据**：
- 从ZIP提取时发现image12只有802 bytes
- OCR识别返回 `{"支出金额":0}`
- 这不是代码bug，而是Excel文件中的图片本身有问题

## 修复方案

### 修复1: 优化OCR提示词和模型配置

**文件**: `src/lib/ocr-service.ts`

1. **增加抖音店铺月度数据截图的缓存完整性检查**（L340附近）：
   ```typescript
   // 店铺月度数据截图：只需要 amounts（金额为0也是有效结果）
   if (platform === '抖音' && imageType === '店铺月度数据截图') {
     return !!(cached.amounts && Object.keys(cached.amounts).length > 0);
   }
   ```

2. **优化 buildDouyinShopMonthlyPrompt 提示词**（L567附近）：
   - 增加更明确的金额识别指引
   - 强调"必须识别出具体的金额数值，不要返回0除非确实为0"
   - 增加示例说明

3. **增加OCR结果验证**（tryModel方法中，L286附近）：
   - 如果OCR返回的金额全为0，但图片大小正常（>10KB），标记为"可能识别不准确"
   - 返回结果时附带警告信息，但不阻止流程

### 修复2: 增加图片质量检查

**文件**: `src/lib/platforms/douyin.ts`

1. **在processImage中增加图片大小检查**（L250附近）：
   - 如果图片小于1KB，跳过OCR识别，直接返回null
   - 记录日志：`[抖音] 图片过小(${size} bytes)，跳过OCR识别`

### 修复3: 增加OCR重试机制（针对金额全为0的情况）

**文件**: `src/lib/ocr-service.ts`

1. **在tryModel中增加结果验证**（L286附近）：
   - 如果OCR返回的金额全为0，且图片大小>10KB，自动重试一次
   - 重试时使用更详细的提示词
