# 计划：检查并修复"版本号更新后系统仍继续修改"的问题

## 问题分析

### 现象
CHANGELOG.md 中存在两个 `1.16` 版本条目（第7行和第12行），说明：
1. 第一次完成"版本号更新 + Docker同步 + Git提交"后（修改：仅对比第一个sheet页）
2. 系统又做了额外修改（修复：5个bug），但没有递增版本号，而是追加了第二个 `1.16` 条目

### 根因
项目规则文档 `.trae/rules/README.md` 存在以下约束缺失：

| 缺陷 | 说明 |
|------|------|
| **无"版本封版"规则** | 流程是线性的5步，但没有明确说"Git提交后，该版本封版，禁止再修改" |
| **无"新变更=新版本"规则** | 如果提交后需要额外修改，文档没有强制要求递增版本号 |
| **无"禁止重复版本号"规则** | CHANGELOG.md 允许出现同一版本号的多条记录 |
| **无"提交前完整性检查"** | 没有要求在更新版本号之前确认所有修改已完成 |

### 当前流程的问题
```
修改代码 → 验证 → 更新版本号 → 同步Docker → Git提交
                                                    ↓
                                            系统又发现需要修改
                                                    ↓
                                            直接修改代码（未回到步骤1）
                                                    ↓
                                            再次更新版本号？→ ❌ 没有规则要求
```

## 修改方案

### 修改文件
[.trae/rules/README.md](file:///d:/trea项目/多平台账单对比系统/.trae/rules/README.md)

### 具体修改内容

#### 1. 核心原则新增第4条（约第6行后）
新增"版本封版"核心原则：
```
4. **Git 提交后即版本封版，该版本禁止再有任何代码修改；如需修改必须递增版本号重新走完整流程**
```

#### 2. 开发修改流程新增"封版规则"（约第64行后，Step 5之后）
在步骤表格后新增封版规则说明：
```markdown
**封版规则（Git 提交后强制执行）**：

| 规则 | 说明 |
|------|------|
| 禁止同版本二次修改 | Git 提交后，该版本代码封版，禁止再修改任何文件 |
| 新变更必须新版本 | 提交后发现需要修改，必须递增版本号，重新走完整 5 步流程 |
| 禁止重复版本号 | CHANGELOG.md 中同一版本号只能出现一次 |
| 提交前确认完整性 | 更新版本号前，必须确认所有代码修改已完成，不再有遗漏 |
```

#### 3. 禁止事项新增（约第182行后）
在 Docker 禁止事项后新增：
```markdown
- ❌ 禁止 Git 提交后对已提交版本再做代码修改（必须递增版本号重新走流程）
- ❌ 禁止 CHANGELOG.md 中同一版本号出现多条记录
```

#### 4. 流程门禁新增第7步（约第275行后）
在流程门禁表格末尾新增：
```
| 7 | **版本封版** | **Git 提交后该版本封版，如需修改必须递增版本号重新走完整流程** |
```

## 验证步骤

1. 读取修改后的 README.md，确认4处修改均已生效
2. 确认核心原则有4条
3. 确认封版规则表格存在
4. 确认禁止事项新增2条
5. 确认流程门禁有7步
6. 运行版本号更新流程（当前版本1.16 → 1.17）
7. 同步 Docker
8. Git 提交推送

## 假设与决策

| 项目 | 决策 | 理由 |
|------|------|------|
| 是否修复 CHANGELOG.md 中的重复1.16 | 是，合并为一条记录 | 同版本号不应有两条记录 |
| 版本号是否递增 | 是，1.16 → 1.17 | 本次修改了规则文档，属于新变更 |
| 封版规则的触发点 | Git commit 完成时 | commit 是版本封版的自然分界点 |
