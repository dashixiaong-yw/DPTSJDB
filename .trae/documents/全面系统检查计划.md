# 全面系统检查计划 — 多平台账单对比系统

## 概述

对系统进行全面代码审查，发现并修复所有影响系统正确运行的 Bug、逻辑错误和潜在问题。

---

## 当前状态分析

### 系统架构
- **前端**: Next.js App Router + React + Tailwind CSS + shadcn/ui
- **后端**: Next.js API Routes（内存存储，无数据库）
- **OCR**: 通过 OpenAI SDK 调用 Kimi K2.5 视觉模型
- **平台**: 抖音、拼多多、淘宝三个平台处理器
- **存储**: 本地文件系统（data/uploads/）

### 代码规模
- 核心库文件: 8个（comparison-engine.ts, excel-parser.ts, ocr-service.ts, task-processor.ts, memory-store.ts, local-storage.ts, services.ts, utils.ts）
- 平台处理器: 4个（base.ts, douyin.ts, pinduoduo.ts, taobao.ts）
- API路由: 10个
- 前端页面: 2个（首页、结果页）

## 发现的问题（按严重程度排序）

### P0 — 关键 Bug（系统无法正常运行）

#### 1. 抖音处理器图片查找逻辑完全失效
- **文件**: [douyin.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/platforms/douyin.ts) L266-278
- **问题**: `getImageByColIndex` 方法遍历 `imagesForRow` Map，将 key 当作 cellRef 来匹配列字母。但 `buildRowImagesMap`（start/route.ts L230-248）使用 `imageType` 作为 key（如"店铺月度数据截图"），而非 cellRef。正则 `/^[A-Z]+/` 无法匹配中文字符，导致**抖音平台永远找不到图片，无法进行任何比对**。
- **影响**: 抖音平台完全无法工作
- **修复**: 改用 `imagesForRow.get('店铺月度数据截图')` 和 `imagesForRow.get('支出总额截图')` 直接按 imageType 获取图片，与拼多多、淘宝处理器保持一致

#### 2. 内存存储自旋锁导致死锁
- **文件**: [memory-store.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/memory-store.ts) L84-88
- **问题**: `acquireLock` 使用 `while (resultStoreLocks.get(taskId))` 自旋等待。Node.js 是单线程的，如果锁被持有，没有其他代码可以执行来释放锁，导致**永久死锁**。
- **影响**: 当多个比对结果同时追加到同一任务时，系统会卡死
- **修复**: 移除自旋锁，改用简单的同步操作（Node.js 单线程下 `appendTaskResults` 本身是原子的，不需要锁）

#### 3. saveResults 缺少 is_zero_value 字段
- **文件**: [start/route.ts](file:///d:/trea项目/多平台账单对比系统/src/app/api/task/[taskId]/start/route.ts) L301-320
- **问题**: `saveResults` 函数映射 `ComparisonItem` 到 `ComparisonRecord` 时，遗漏了 `is_zero_value` 字段。淘宝处理器设置了 `isZeroValue: true`，但保存时丢失。
- **影响**: 结果页面"无需核对"标记无法显示
- **修复**: 在 records 映射中添加 `is_zero_value: item.isZeroValue || null`

### P1 — 重要问题（功能异常或数据不一致）

#### 4. BUILTIN_FIELD_MAPPINGS 重复定义且不一致
- **文件**: [comparison-engine.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/comparison-engine.ts) L41-76 vs [base.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/platforms/base.ts) L14-59
- **问题**: 字段映射在两个文件中分别定义，且内容不一致。例如 `comparison-engine.ts` 中 `'成交金额': '成交金额'`，而 `base.ts` 中 `'成交金额': '成交额'`。如果旧代码路径被意外调用，会产生错误的映射。
- **影响**: 潜在的数据比对错误
- **修复**: 删除 `comparison-engine.ts` 中的重复定义，统一使用 `base.ts` 中的版本

#### 5. ComparisonEngine 类是死代码
- **文件**: [comparison-engine.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/comparison-engine.ts) L188-961
- **问题**: `ComparisonEngine` 类的 `compare()`、`compareRow()`、`comparePDDRow()` 方法已不被任何代码调用。新的平台处理器架构使用 `base.ts` 中的工具函数和各平台处理器。这个 964 行的文件中大部分是死代码。
- **影响**: 代码维护混乱，容易误用旧接口
- **修复**: 删除 `ComparisonEngine` 类及其相关代码，只保留 `getResults` 方法（如果仍被使用）或将其迁移到其他位置

#### 6. getImageType 函数重复定义
- **文件**: [excel-parser.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/excel-parser.ts) L54-103 vs [base.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/platforms/base.ts) L267-316
- **问题**: 完全相同的 `getImageType` 函数在两个文件中定义
- **影响**: 维护时容易遗漏一处，导致行为不一致
- **修复**: `excel-parser.ts` 中导入 `base.ts` 的 `getImageType`，删除重复定义

#### 7. identifyPlatform 函数重复定义
- **文件**: [excel-parser.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/excel-parser.ts) L893-923 vs [platforms/index.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/platforms/index.ts) L70-86
- **问题**: `excel-parser.ts` 中的 `identifyPlatform` 使用 Jaccard 相似度算法，而 `platforms/index.ts` 中的使用处理器 `identify()` 方法。两者可能产生不同的识别结果。
- **影响**: 平台识别可能不一致
- **修复**: 删除 `excel-parser.ts` 中的 `identifyPlatform`，统一使用 `platforms/index.ts` 的版本

#### 8. ComparisonItem 接口重复定义
- **文件**: [comparison-engine.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/comparison-engine.ts) L5-29 vs [platforms/types.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/platforms/types.ts) L13-36
- **问题**: 两个文件定义了同名但不同的 `ComparisonItem` 接口。`types.ts` 版本有 `isZeroValue` 字段，`comparison-engine.ts` 版本没有。
- **影响**: 类型不一致可能导致编译或运行时问题
- **修复**: 删除 `comparison-engine.ts` 中的 `ComparisonItem`，统一使用 `types.ts` 的版本

#### 9. 下载标记文件 cellRef 为空时无法匹配
- **文件**: [download/route.ts](file:///d:/trea项目/多平台账单对比系统/src/app/api/task/[taskId]/download/route.ts) L46-49
- **问题**: 结果映射使用 `${sheetName}!${cellRef}` 作为 key，但店铺名称和月份比对项的 `cellRef` 为空字符串。这些项无法在下载的 Excel 中被高亮标记。
- **影响**: 下载的标记文件中，店铺名称和月份比对结果不会被高亮
- **修复**: 对于 cellRef 为空的项，使用 `rowIndex` 和 `colIndex` 计算出正确的 cellRef

#### 10. taobao.ts colIndexToRef 不支持超过 Z 列
- **文件**: [taobao.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/platforms/taobao.ts) L388-391
- **问题**: `String.fromCharCode(65 + colIndex)` 只支持 A-Z（0-25），超过 25 的列索引会产生非字母字符。淘宝表格可能有超过 26 列。
- **影响**: 超过 Z 列的字段会生成错误的 cellRef
- **修复**: 使用 `excel-parser.ts` 中已有的 `columnToLetter` 函数（支持 AA、AB 等多字母列）

### P2 — 中等问题（健壮性和一致性）

#### 11. OCR 缓存 result_json 类型不一致
- **文件**: [memory-store.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/memory-store.ts) L55 vs [ocr-service.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/ocr-service.ts) L736-744
- **问题**: `result_json` 类型为 `unknown`，但存储时直接传入 `OCRResult` 对象，读取时尝试 `JSON.parse`（如果是字符串）或直接类型断言。存储和读取逻辑不一致。
- **修复**: 统一为存储时 `JSON.stringify`，读取时 `JSON.parse`

#### 12. JSON 导出时区处理不正确
- **文件**: [result/[taskId]/page.tsx](file:///d:/trea项目/多平台账单对比系统/src/app/result/[taskId]/page.tsx) L208
- **问题**: `new Date(Date.now() + 8 * 60 * 60 * 1000).toISOString()` 是错误的北京时间转换方式，不处理夏令时和时区边界
- **修复**: 使用 `toLocaleString('zh-CN', { timeZone: 'Asia/Shanghai' })` 或专用时区库

#### 13. 分片上传内存泄漏风险
- **文件**: [chunk-v2/route.ts](file:///d:/trea项目/多平台账单对比系统/src/app/api/upload/chunk-v2/route.ts) L11-18
- **问题**: `chunkStoreV2` 存储在模块级变量中，未完成的上传会在内存中保留 30 分钟。大文件（接近 100MB）的分片会占用大量内存。
- **修复**: 添加最大并发上传数限制，或减小超时时间

#### 14. OCR 服务无重试机制
- **文件**: [ocr-service.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/ocr-service.ts) L134
- **问题**: API 调用失败时直接返回错误结果，没有重试机制。网络抖动可能导致整行数据比对失败。
- **修复**: 添加 2-3 次重试，带指数退避

#### 15. 内存存储无大小限制
- **文件**: [memory-store.ts](file:///d:/trea项目/多平台账单对比系统/src/lib/memory-store.ts) L71-74
- **问题**: `taskStore`、`resultStore`、`ocrCacheStore` 没有大小限制，长时间运行可能导致内存溢出
- **修复**: 添加最大条目数限制，超出时自动清理最旧的数据

## 假设与决策

1. **ComparisonEngine 类处理**: 假设该类已完全被平台处理器架构替代，可以安全删除。如果仍有外部引用，需要先迁移。
2. **抖音处理器修复策略**: 选择与拼多多/淘宝一致的 imageType 键查找方式，而非修复 `getImageByColIndex`，因为这是更简单、更一致的方案。
3. **自旋锁移除**: Node.js 单线程环境下，同步的 Map 操作天然是原子的，不需要锁保护。如果未来引入 Worker 线程，需要重新评估。
4. **修复顺序**: 先修复 P0 关键 Bug，再清理重复代码，最后做健壮性改进。每个阶段独立验证。
